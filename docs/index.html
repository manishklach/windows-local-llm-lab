If you've ever tried to squeeze more tokens per second out of a local LLM on a Windows laptop, you know how noisy the process usually is: change a setting, run once, eyeball the number, repeat. windows-local-llm-lab takes the opposite approach — warmups, repeated runs, standard deviation, and a hard rule that nothing is allowed to cook the laptop in the name of a few extra tokens per second.

It's a PowerShell-only repo built around Ollama on native Windows, with parallel tracks exploring llama.cpp directly and WSL. What stands out is how honest it is about what's actually been proven versus what's still an open question.

The setup

All numbers so far come from one clearly documented machine — a modest, CPU-bound laptop, not a workstation with a discrete GPU. That's exactly why the tuning questions matter here: every thread, context length, and batch size choice has a real, measurable effect.

  • CPU — AMD Ryzen 3 5300U, 4 cores / 8 threads
  • Memory — 16 GB RAM
  • Graphics — Integrated AMD Radeon Graphics
  • OS / runtime — Windows 11, Ollama 0.30.11

What's been proven

The README leads with a "current conclusion" section — a nice pattern in itself, since you don't have to dig through scripts to find out what actually works.

  • Native Windows + Ollama remains the fastest safe, proven path on this hardware.
  • The best validated sweep cell lands at 7.62 median eval tok/s — 6 threads, 1024 context, batch 64, on the qwen35-4b-q4km reference model.
  • gemma:2b tops out around 16.00 median eval tok/s at 12 threads — a clean illustration of how much model size alone dominates throughput on constrained hardware.
  • A direct llama.cpp CPU run at 8 threads hit ~6.12 gen tok/s; pushing to 10 threads regressed to 5.47 — oversubscription hurting a 4-core chip once it passes its physical core count.
More threads is not automatically more speed. On a 4-core/8-thread part, the lab's own data puts the sweet spot around 6–12 threads — logical SMT threads don't add real compute, they just add scheduling overhead.

What's still open

To its credit, the lab doesn't paper over its unresolved threads:

OPEN

WSL connectivity — WSL itself is healthy, but the Windows-hosted Ollama API isn't currently reachable from inside it, so there are no valid WSL-vs-native numbers yet.

OPEN

Vulkan — llama-bench doesn't currently see an available Vulkan device, so the integrated GPU path — likely the single biggest untapped throughput lever here — hasn't been benchmarked at all.

Both are flagged plainly as blockers rather than glossed over — exactly what you want from a benchmarking repo: a clear line between "measured" and "not yet possible to measure."

Safety as a first-class citizen

A lot of local-LLM tuning content quietly assumes you're fine running unstable overclocks or maxing out every setting to chase a benchmark win. This repo goes the other way, with explicit rules baked into the scripts themselves:

Hard rules
  • No Realtime process priority
  • No disabling thermal protections
  • No benchmarking on battery — stay plugged in
  • Laptop on a hard surface (airflow matters)
  • Stop immediately if the machine reboots repeatedly
  • Prefer 128–256 generated tokens for real measurements; short runs are smoke tests only

The enter-max-perf-mode.ps1 / exit-max-perf-mode.ps1 pairing is a particularly nice touch — whatever tuning gets applied for a benchmark run is fully reversible afterward, rather than leaving the system in some semi-optimized state you forget about a week later.

Boosting throughput on hardware like this

Pulling together the lab's own measured takeaways, in order of expected impact:

  1. Cap threads near your physical core count, not your logical one. 6–12 threads outperforms 16 or 32 every time in this data — oversubscription is a real, measurable tax on CPU-bound decode.
  2. Model size still wins big. gemma:2b more than doubled the throughput of the 4B reference model. If your task tolerates a smaller model, that's a bigger lever than any thread or batch tweak.
  3. Keep context length only as large as you need. Shorter-context results were consistently competitive with or faster than longer ones — every extra token of context eats memory bandwidth that could go toward generation.
  4. Chase the still-unproven Vulkan path. For integrated-GPU laptops, offloading matmuls to the iGPU is usually the single biggest available throughput jump once device visibility is sorted out.
  5. Measure, don't guess. Multiple runs, warmups, and reported standard deviation are what make any of the above conclusions trustworthy. A single benchmark run on a laptop is basically noise.

Why it's worth a look

This isn't trying to be a definitive local-LLM leaderboard — it's a working lab notebook for one specific laptop, with the discipline to admit what hasn't been figured out yet. That's more useful than a polished-looking benchmark chart with no methodology behind it. If you're tuning local inference on similar Windows hardware, the tools/ scripts and the safe-sweep pattern are a solid fork-and-point starting point for your own machine.
